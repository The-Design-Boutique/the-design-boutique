# Ruleset: Rank Math Pro Style Toolset (SOW 2.5)

Each feature has its own rules and its own definition of done. The SOW includes the full set in this trial; the definitions of done are the fence against sprawl.

## 1. On-page SEO score and checklist

1. The score component lives in the document editor, recomputes on edit (debounced), and needs no network round-trip: all checks run on the document value client-side.
2. Checks and weights (100 total): focus keyword present in SEO title (15), in first ~10% of body (10), in meta description (10), in slug (5), in at least one h2/h3 (5), keyword density between 0.5% and 2.5% (10), title length within guidance (10), meta description length within guidance (10), all images have alt text (10), at least one internal link (10), readability band at least "ok" (5).
3. Bands: 80+ good (green), 50 to 79 needs work (amber), below 50 poor (red). Mirrors Rank Math's familiar coloring so the team recognizes it.
4. No focus keyword set: keyword checks show as skipped, score computes over the remaining weights, and the checklist nudges the editor to set one. Never punish an unset keyword with a zero.
5. The check implementations are shared with the local check suite from ruleset 03, rule 16. One library.

## 2. Readability analysis

6. Computed from Portable Text on the same cadence as the score. Flags: sentences over 25 words, paragraphs over 150 words, consecutive sentences starting with the same word (3+), passive voice occurrences (auxiliary + past participle heuristic; accept imperfect detection, it is advisory).
7. Output is a band (good / ok / needs work) plus the specific flagged sentences, so editors can click through rather than guess.

## 3. XML sitemap

8. Generated by the frontend from published, routable documents. Excluded: anything with robots.index false, redirects, operational documents, unpublished drafts.
9. `lastmod` from the document's `_updatedAt`. Split into a sitemap index if any single sitemap would exceed 5,000 URLs (it will not for this site; the rule exists so the code does not hard-fail later).
10. While staging: the sitemap builds and is verifiable, but the site-wide noindex stands. Do not submit the staging sitemap anywhere.

## 4. Redirect manager, 404 monitor, redirect log

11. `redirect` document type: `fromPath` (unique, leading slash, no domain), `toPath` or reference to a page, `statusCode` (301 default, 302 allowed), `enabled`, `notes`, `createdBy`, `hitCount`, `lastHitAt`.
12. Publishing a slug change on a live-routable document prompts creation of a 301 from the old path; accepting is one click and the default. Declining requires the editor to actively opt out. Chains are flattened on write: if A points to B and B moves to C, A is rewritten to point at C.
13. Middleware consults the redirect set on 404-bound requests. Redirect loops are structurally prevented on write (a redirect may not target a `fromPath` of another enabled redirect after flattening; self-reference blocked).
14. 404 monitor: the not-found handler logs path, referrer, timestamp, count-aggregated per path per day into `notFoundEntry` documents. Cap: 1,000 distinct paths; beyond that, lowest-count oldest entries are pruned by the daily cron. Static-asset noise (favicon, .well-known, wp- probe paths) is filtered before logging.
15. Studio tool lists 404s by frequency with a one-click "create redirect" action per row. That is the whole loop: see the 404, fix it in place.

## 5. Canonical and Open Graph fallbacks

16. Canonical precedence: explicit `seo.canonicalUrl` if set, else the page's own absolute staging URL (self-canonical). Written once as a pure function used by every route.
17. OG precedence: `ogTitle` -> `seo.title` -> page title; `ogDescription` -> `metaDescription` -> excerpt if the type has one; `ogImage` -> page featured image -> site default share image (a required field in site settings). Twitter falls back to OG, then the same chain. No page can render without complete OG tags.

## 6. Local SEO (SF office)

18. A singleton `officeLocation` settings document: name, address fields, geo, phone, email, hours, sameAs profile URLs. Rendered as LocalBusiness JSON-LD on the contact page (and any page whose `schemaType` is LocalBusiness), assembled from structured fields, never hand-written JSON.

## 7. Content writing assistance (hybrid, decided July 21, 2026)

19. Always on: the deterministic layer, which is the score checklist plus readability plus heading-structure hints (missing h2s, skipped heading levels, keyword absent from headings). No network, no cost, no configuration.
20. Optional AI layer: a "suggest" control in the assist panel that appears only when the server reports a configured API key. Requests run server-side through one route; the key never reaches the browser. Suggestions are inserted only by explicit editor action, never automatically.
21. Before enabling the AI layer for the client: record whose key and whose budget in this file. Until then it ships dark. This is the one feature in 2.5 with a per-use cost; it must not silently run on someone's dime.

## 8. SEO Health panel (was: Semrush sync)

22. The SOW originally listed Semrush sync here; it is replaced by the free SEO Health panel and governed entirely by ruleset 03 (`03-ruleset-seo-health-panel.md`). Semrush is kept out of this project. This section exists so nobody implements the panel twice: the sources, placement, and refresh rules there are the single source of truth. The in-CMS check library is shared between that panel and the on-page score in section 1 above.

## 9. Historical CWV trending

23. Data: the `cwvSnapshot` documents from ruleset 02, seeded once from the CrUX History API (about 40 weeks of weekly points, flagged `seeded: true`) and extended daily thereafter.
24. Charts live in the CWV dashboard tool: per metric, per form factor, value line with Google's threshold bands shaded behind it, range selectable (30/90 days, all). Weekly seeded points and daily live points may coexist on one chart; the granularity shift is labeled, not smoothed over.
25. Trend direction (improving/stable/degrading over the selected range) is stated per metric in plain text, because that is the sentence Laney's team will actually read.

## Trial-wide definition of done for 2.5

Every numbered feature above demonstrably works on staging during the walkthrough, using real thedesignboutique.com content, and each has its rules here honored exactly. Anything beyond these definitions (more checks, more charts, more automation) is post-trial conversation, not silent scope growth.
